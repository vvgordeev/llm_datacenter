# Архитектура LLM Datacenter

Обзор для работы над любой задачей. Детали конкретной области — в `docs/features/*`. Метод — `CONVENTIONS.md`.

## Слои

```
Сбор (скилл /digest: WebSearch/WebFetch + scripts/gemini_video.py, extract_frames.py)
  → Данные (data/daily/*.json — двуязычные, data/index.json, data/state.json — дедуп)
  → Билд (scripts/build_site.py: data/ → docs/data/, генерация конспект-страниц docs/notes/)
  → Публикация (git push → GitHub Pages из docs/)
  → Просмотр (docs/index.html — SPA без сборки: рендер, срезы Неделя/Месяц, фильтры, i18n, темы)
```

Управление сбором — `config/sources.yaml` (правит пользователь). Фильтры отображения — на клиенте (localStorage), на сбор не влияют.

## Инварианты (нельзя ломать)

1. **Дневные JSON append-only:** `data/daily/<date>.json` после публикации не редактируются задним числом (кроме исправления брака за сегодня). Архив — источник истины для срезов.
2. **Все текстовые поля данных двуязычны** — объект `{ru, en}`. UI не содержит захардкоженных строк: словарь интерфейса в `docs/assets/app.js` (i18n), контент — из JSON.
3. **Каждый факт — со ссылкой.** Новость, мнение, кейс, тезис конспекта ссылаются на первоисточник. Материал без URL в данные не попадает.
4. **`impact_score` (1–10) обязателен** у каждой новости + однострочное обоснование по рубрике (см. `.claude/skills/digest/SKILL.md`). Сортировка и срезы Неделя/Месяц строятся только на нём и `cluster_id`.
5. **Дашборд — статический:** никакого бэкенда/БД; недельные и месячные срезы считаются на клиенте из дневных JSON. Внешние запросы только к своим `docs/data/*`.
6. **Секреты** (GEMINI_API_KEY и др.) — только в переменных окружения; в коде, данных и git их нет.
7. **Демо-данные помечены** `"demo": true` и показываются с баннером; реальный прогон их заменяет, а не дополняет.

## Матрица доступа

Публичный статический сайт, авторизации нет. Записывает данные только пайплайн (routine/скилл с push-доступом к репо); читатель — любой.

## Модели данных

`data/daily/<date>.json`:
- `date`, `generated_at`, `demo?`
- `news[]`: `id`, `cluster_id` (сюжет для склейки follow-up'ов в срезах), `models[]` (теги компаний/моделей из sources.yaml), `title{ru,en}`, `summary{ru,en}`, `impact_score` (1–10), `impact_reason{ru,en}`, `source_url`, `opinions[]` (`author`, `stance` positive|skeptical|neutral, `quote{ru,en}`, `url`), `use_cases[]` (`text{ru,en}`, `url`)
- `expert_content[]`: `id`, `type` article|video, `author`, `title{ru,en}`, `url`, `summary{ru,en}`, `note_page` (путь в docs/notes/), `cover` (кадр-обложка карточки, опционально)

`data/index.json`: список дней с метаданными (дата, число новостей, топ-заголовок). `data/state.json`: **опубликованные** URL (только source_url новостей и конспекты — не вспомогательные ссылки) + cluster_id → скользящее окно 60 дней. `data/run_log/<date>.json`: отчёт о сборе за прогон (статусы источников, счётчики, статус видео-пайплайна) — для диагностики, на сайт не копируется.

## Сквозные паттерны

- **Ранжирование** — только через рубрику impact_score в SKILL.md; менять рубрику = ADR.
- **Срезы Неделя/Месяц:** взять дни периода → сгруппировать по `cluster_id` → в кластере оставить свежайшую запись с максимальным score → топ-N по score.
- **Конспект видео:** Gemini возвращает тезисы + таймкоды + список «значимых демонстраций»; кадры извлекаются только по этим таймкодам (≤6, jpg).
- **Пути:** скрипты не полагаются на ASCII-пути (кириллица в пути проекта) — везде pathlib и UTF-8.
