Полный ИИ-стек 2026: мультиагентная архитектура на MCP, A2A и AG-UI The complete AI tech stack in 2026: a multi-agent architecture on MCP, A2A and AG-UI

Канал AI Explained собирает работающий мультиагентный планировщик путешествий, чтобы на конкретном примере показать, из каких слоёв в 2026 году собирают production-стек вокруг LLM [00:00]. Ключевой архитектурный выбор задан сразу [00:16]: вместо одного большого агента-монолита — микросервисная архитектура, где каждый агент работает в собственном контейнере. Это даёт изоляцию сбоев (падение одного агента не роняет систему), модульность (чёткие контракты между сервисами, никакого общего состояния) и гибкость — модели, векторные хранилища и слои памяти можно заменять по отдельности, не переписывая систему целиком. AI Explained assembles a working multi-agent trip planner to use a concrete example for showing what layers a production LLM stack is built from in 2026 [00:00]. The key architectural choice is stated right away [00:16]: instead of one large monolithic agent, a microservice architecture where each agent runs in its own container. That buys failure isolation (one agent crashing doesn't take down the system), modularity (clear contracts between services, no shared state), and flexibility — models, vector stores, and memory layers can each be swapped independently without rewriting the whole system.

Три протокола вместо самодельной шины сообщенийThree protocols instead of a homegrown message bus

Вместо того чтобы изобретать формат обмена сообщениями между частями системы, видео показывает три специализированных протокола, каждый под свою задачу [01:08]: A2A (Agent-to-Agent) — для задач между агентами; MCP (Model Context Protocol) — для обращения агента к внешним инструментам; AG-UI (Agent-to-UI) — для потоковой трансляции живого прогресса работы агента на фронтенд. Структура самих агентов строится на LangGraph вокруг связки «оркестратор + специализированные агенты» [02:16]: оркестратор разбирает запрос пользователя и раздаёт подзадачи специалистам (поиск рейсов, отелей, маршрутов), а не пытается сделать всё сам одним проходом. Rather than inventing a custom message format between the system's parts, the video shows three specialized protocols, each scoped to its own job [01:08]: A2A (Agent-to-Agent) for inter-agent tasks; MCP (Model Context Protocol) for an agent talking to external tools; AG-UI (Agent-to-UI) for streaming an agent's live work progress to the frontend. The agents themselves are structured on LangGraph around an "orchestrator plus specialist agents" pattern [02:16]: the orchestrator parses the user's request and hands out subtasks to specialists (flight search, hotel search, routing) instead of trying to do everything itself in one pass.

Инструменты как сервисы, память — отдельным слоемTools as services, memory as its own layer

Дальше видео разводит по разным сервисам две вещи, которые в наивных агентных реализациях часто смешивают в одном процессе. Инструменты (поиск, погода, бронирование) реализованы не как встроенные функции внутри агента, а как независимые веб-сервисы, говорящие по MCP [03:06] — тогда любой агент или фреймворк может обнаружить и использовать инструмент одинаковым способом. Память вынесена в отдельный сервис поверх Mem0 [03:31]: он извлекает факты из разговоров и предпочтения пользователя и отдаёт их как контекст для будущих обращений, но писать в память разрешено только оркестратору — иначе разные агенты, не зная друг о друге, начинают записывать противоречивые факты. Хранилище тоже подобрано под задачу: PostgreSQL с расширением pgvector — для RAG-контента и состояния разговора, а отдельная высокопроизводительная векторная база Qdrant — специально под долгосрочную память сервиса Mem0. The video then splits into separate services two things that naive agent implementations often bundle into one process. Tools (search, weather, booking) are implemented not as functions embedded in the agent but as independent web services speaking MCP [03:06] — so any agent or framework can discover and use a tool the same way. Memory is pulled out into its own service on top of Mem0 [03:31]: it extracts facts from conversations and user preferences and hands them back as context for future turns, but only the orchestrator is allowed to write to it — otherwise separate agents, unaware of each other, start writing contradictory facts. Storage is likewise matched to the job: PostgreSQL with the pgvector extension for RAG content and conversation state, and a separate high-performance vector database, Qdrant, specifically for the Mem0 service's long-term memory.

Сквозная трассировка вместо логов вручнуюEnd-to-end tracing instead of hand-rolled logging

Последний слой стека — наблюдаемость [04:19]: каждый пользовательский запрос порождает трассировку через OpenTelemetry, визуализируемую в Langfuse, охватывающую все сервисы, агентов, инструменты и вызовы модели — задержки, ошибки, расход токенов и метаданные видны как единая цепочка, а не разрозненные логи из разных контейнеров. Финал видео — живая демонстрация всей системы в работе: планировщик путешествий получает запрос, оркестратор раздаёт его специализированным агентам, а зритель видит в реальном времени, как каждый протокол и сервис отрабатывает свою часть [06:01]. The stack's last layer is observability [04:19]: every user request generates a trace via OpenTelemetry, visualized in Langfuse, spanning all services, agents, tools, and model calls — latency, errors, token spend, and metadata show up as a single chain rather than scattered logs from different containers. The video closes with a live demo of the whole system running: the trip planner receives a request, the orchestrator hands it to specialist agents, and the viewer watches in real time as each protocol and service does its part [06:01].
