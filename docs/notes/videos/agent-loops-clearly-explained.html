Нейт Херк начинает с твита Питера Штайнбергера [00:08] о том, что главный сдвиг в работе с агентами — это переход от написания промптов к проектированию циклов: вместо одной инструкции для одного ответа вы задаёте цель и правило проверки, а агент сам крутит цикл «рассуждение → действие → наблюдение» до тех пор, пока условие не будет выполнено [00:45]. Nate Herk opens with a tweet from Peter Steinberger [00:08] arguing that the real shift in working with agents is moving from writing prompts to designing loops: instead of one instruction for one answer, you set a goal and a verification rule, and the agent itself cycles through reason → act → observe until the condition is met [00:45].

Главное условие: объективный критерий «готово»The core requirement: an objective "done" criterion

Херк подчёркивает: цикл работает только тогда, когда у ИИ есть чёткий, объективно проверяемый способ понять, что задача выполнена — субъективная оценка «выглядит нормально» не подходит, потому что агент либо остановится слишком рано, либо будет крутиться бесконечно [10:37]. На графике он показывает, как качество результата растёт с каждой итерацией цикла именно потому, что каждая итерация сверяется с этим критерием, а не идёт вслепую [03:04]. Herk stresses that a loop only works when the AI has a clear, objectively checkable way to know the task is done — a subjective "looks fine" won't do, because the agent will either stop too early or spin forever [10:37]. A chart shows quality climbing with each loop iteration precisely because every iteration checks itself against that criterion instead of running blind [03:04].

Когда цикл вообще нужен, и какие бывают архитектурыWhen a loop is actually warranted, and what architectures exist

Дальше — практическое руководство по принятию решения [05:26]: агентный цикл нужен не для любой задачи, а именно для повторяющихся работ с непредсказуемым числом шагов, где результат можно объективно проверить программно или моделью-судьёй. Херк перечисляет архитектуры по возрастанию сложности — одиночный агент, который сам себя проверяет; связка «создатель-проверяющий», где один агент делает, а второй независимо оценивает; и иерархия «менеджер-помощники», где верхнеуровневый агент раздаёт подзадачи нескольким исполнителям. Выбор архитектуры зависит от того, насколько задачу можно разложить на независимо проверяемые куски. Next comes a practical decision guide [05:26]: an agent loop isn't for every task — it's specifically for repetitive work with an unpredictable number of steps, where the result can be objectively checked by code or a judge model. Herk lays out architectures in order of complexity — a solo agent that checks its own work; a "maker-checker" pair where one agent does the work and a second independently evaluates it; and a "manager-helpers" hierarchy where a top-level agent hands out subtasks to several workers. Which architecture fits depends on how cleanly the task splits into independently verifiable pieces.

Демонстрация на реальной задаче и предостережение от переусложненияA live demo, and a caution against over-engineering

Как иллюстрацию Херк показывает цикл генерации миниатюр для видео: агент создаёт вариант обложки, сверяет его с критериями (читаемость текста, контраст, соответствие теме), при необходимости переделывает — и так до прохождения проверки [06:28]. В конце — личный опыт и предостережение [12:24]: циклы стоят денег и времени при каждой итерации, поэтому нужны жёсткие ограничения (максимум итераций, бюджет по стоимости) и постоянный контроль, оправдывает ли качество результата эти траты; не всем задачам и не всем людям нужны круглосуточные автономные агенты — часто короткий, узко сфокусированный цикл продуктивнее, чем попытка автоматизировать всё сразу. As an illustration, Herk walks through a thumbnail-generation loop: the agent produces a cover draft, checks it against criteria (text legibility, contrast, on-topic fit), redoes it if needed — and repeats until it passes [06:28]. He closes with personal experience and a caution [12:24]: loops cost money and time on every iteration, so they need hard limits (a max iteration count, a cost budget) and constant monitoring of whether the quality gain is worth the spend; not every task or every person needs a 24/7 autonomous agent — a short, narrowly scoped loop is often more productive than trying to automate everything at once.
