Как писать хорошие агентные скиллы: недостающее руководство Building Great Agent Skills: The Missing Manual

Скиллы для агентов (переиспользуемые инструкции вроде SKILL.md для Claude Code) расплодились настолько, что Мэтт Покок описывает это состояние как «ад навыков» [00:50]: пользователи и команды не могут отличить хороший скилл от плохого и просто нагромождают их один на другой. Ответ — не «ещё один скилл», а чек-лист из четырёх пунктов для оценки и улучшения того, что уже написано: Триггер, Структура, Управление, Обрезка [02:20]. Agent skills (reusable instruction files like SKILL.md for Claude Code) have multiplied to the point Matt Pocock calls it "Skill Hell" [00:50]: users and teams can't tell a good skill from a bad one and just keep piling more on. His answer isn't "write one more skill" but a four-point checklist for evaluating and improving what already exists: Trigger, Structure, Steering, Pruning [02:20].

Триггер: кто решает, что скилл нужен сейчасTrigger: who decides the skill is needed now

Первый пункт — осознанный выбор между двумя способами вызова скилла [05:09]. Вызов пользователем (например, слэш-команда) даёт предсказуемость — скилл срабатывает точно тогда, когда его позвали, — но перекладывает когнитивную нагрузку на человека: нужно помнить, что скилл вообще существует и когда его использовать. Вызов моделью — агент сам решает подключить скилл, ориентируясь на его описание, — снимает нагрузку с пользователя, но добавляет её агенту: чем больше таких скиллов одновременно доступно, тем больше контекста тратится на то, чтобы понять, какой из них подходит, и тем выше риск, что агент выберет не тот или пропустит нужный. The first point is a conscious choice between two ways of invoking a skill [05:09]. User-invoked (a slash command, say) is predictable — the skill fires exactly when called — but shifts the cognitive load onto the human: they have to remember the skill exists and when to reach for it. Model-invoked — the agent decides on its own to pull the skill in, based on its description — removes that load from the user but adds it to the agent: the more such skills are available at once, the more context gets spent figuring out which one applies, and the higher the risk the agent picks the wrong one or misses the right one.

Структура: шаги против справочника, и маленький SKILL.mdStructure: steps vs. reference, and a small SKILL.md

Второй пункт — как организовать содержимое скилла [08:44]: разделять «шаги» (собственно процедуру — что делать по порядку) и «справочную информацию» (вспомогательные детали, нужные не всегда). Главный файл SKILL.md должен оставаться как можно меньше — ветвящийся справочный материал прячется за отдельными файлами, на которые в SKILL.md стоят лишь короткие указатели-ссылки. Так агент подгружает в контекст только то, что реально нужно для текущей задачи, а не весь скилл целиком, — это одновременно снижает стоимость токенов и облегчает поддержку. The second point is how to organize a skill's contents [08:44]: separate "steps" (the actual procedure — what to do, in order) from "reference" (supporting detail that isn't always needed). The main SKILL.md file should stay as small as possible — branching reference material is hidden behind separate files that SKILL.md only points to with short links. That way the agent loads into context only what the current task actually needs, not the whole skill at once — cutting token cost while making the skill easier to maintain.

Управление: ведущие слова направляют рассуждение агентаSteering: leading words guide the agent's reasoning

Покок показывает практический приём: последовательные «ведущие слова» внутри скилла — например, фраза «Вертикальный срез» как обязательный первый шаг любого плана — заметно влияют на то, как агент строит рассуждение и во что превращается план реализации [12:20]. На примере он показывает, как сложную задачу стоит разбивать на более мелкие скиллы — это вынуждает агента делать более тщательную подготовительную работу перед тем, как приступить к реализации, вместо того чтобы сразу бросаться писать код [15:48]. Pocock demonstrates a practical trick: consistent "leading words" inside a skill — for instance, the phrase "Vertical Slice" as a mandatory first step in any plan — measurably shape how the agent reasons and what its implementation plan turns into [12:20]. He shows an example of splitting a complex task into smaller skills, which forces the agent to do more thorough legwork before implementation instead of jumping straight into writing code [15:48].

Обрезка: тест на удалениеPruning: the deletion test

Последний пункт — регулярная чистка: скиллы со временем накапливают «no-ops» (формулировки, которые никак не влияют на поведение агента), «осадок» (нерелевантный или устаревший материал) и повторы. Практический приём — тест на удаление: убрать фрагмент и проверить, изменилось ли поведение агента; если нет — фрагмент не нужен [18:52]. Покок завершает видео призывом применить весь чек-лист к собственным скиллам через отдельный скилл /writing-great-skills, который он же и выпустил [19:55]. The last point is regular cleanup: skills accumulate "no-ops" (wording that doesn't affect agent behavior at all), "sediment" (irrelevant or stale material), and repetition over time. The practical trick is a deletion test: remove a fragment and check whether the agent's behavior changes; if it doesn't, the fragment wasn't needed [18:52]. Pocock closes by urging viewers to run the whole checklist on their own skills via a companion skill he shipped, /writing-great-skills [19:55].
