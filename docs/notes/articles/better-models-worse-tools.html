Модели лучше, инструменты хуже: почему новые Claude всё чаще ломают схему tool-вызова Better Models, Worse Tools: why newer Claudes increasingly break the tool-call schema

Оригинал статьи ↗Original article ↗

Ключевые тезисыKey theses

  1. Наблюдение Армина Ронахера: в стороннем харнессе Pi новые модели Claude (Opus 4.8, Sonnet 5) чаще вызывают edit-инструмент с лишними, выдуманными полями в массиве edits[] — сам edit обычно верный, но аргументы не совпадают со схемой. Armin Ronacher's observation: in the third-party Pi harness, newer Claude models (Opus 4.8, Sonnet 5) more often call the edit tool with extra, invented fields in the edits[] array — the edit itself is usually correct, but the arguments don't match the schema.
  2. Инструмент отклоняет вызов и просит повторить. Модель тратит ход на то, чтобы понять, что её вызов не принят, и переформулировать его в правильном формате. The tool rejects the call and asks to retry. The model burns a turn figuring out its call was rejected and reformatting it correctly.
  3. Парадокс: хуже именно у новых моделей. Ни одна из старых моделей семейства так не делает — регрессия появляется вместе с ростом качества модели, а не вопреки ему. Paradox: it's worse in the newer models specifically. None of the older models in the family show this — the regression arrives alongside the quality improvement, not despite it.
  4. Гипотеза Ронахера: новые модели сильнее дообучены (вероятно, через RL) под канонический edit-инструмент, встроенный в Claude Code. Когда сторонний харнесс предлагает инструмент с тем же смыслом, но другой схемой, более «уверенная» модель сопротивляется сильнее, а не подстраивается легче. Ronacher's hypothesis: newer models are more heavily trained (likely via RL) toward Claude Code's canonical edit tool. When a different harness offers a tool with the same intent but a different schema, the more confident model fights it harder instead of adapting more easily.

Почему это важноWhy it matters

Для разработчиков агентных харнессов вне Claude Code (включая кастомные обвязки над Anthropic API) это предупреждение: рост качества модели по общим бенчмаркам не гарантирует, что она станет удобнее работать с вашей схемой инструментов — может быть наоборот. Это прямое следствие того, как модели вроде Claude Opus 4.8 и Sonnet 5 обучают через RL на конкретных агентных задачах (см. также обновления Claude Developer Platform и рост тарифов на агентную работу в сводках этой недели): оптимизация под один харнесс создаёт скрытую стоимость переноса на другой. For developers building agent harnesses outside Claude Code (including custom wrappers over the Anthropic API), this is a warning: better scores on general benchmarks don't guarantee a model becomes easier to use with your own tool schema — it can go the other way. It's a direct consequence of how models like Claude Opus 4.8 and Sonnet 5 get RL-trained on specific agentic tasks (see also this week's Claude Developer Platform and agentic-tooling updates in the digest): optimizing for one harness creates a hidden cost when porting to another.
