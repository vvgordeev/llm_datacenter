sqlite-utils 4.1: агент добывает лёгкие фичи из трекера задач sqlite-utils 4.1: an agent mines the issue tracker for easy wins

Оригинал статьи ↗Original article ↗

Продолжение саги sqlite-utils (релиз 4.0 «руками Fable» был в наших сводках 7 и 9 июля): минорная версия 4.1, интересная не столько фичами, сколько тем, откуда они взялись. Уиллисон поручил Codex просмотреть открытые issues репозитория и выбрать кандидатов, «лёгких в реализации» — так в релиз попали опция --code (строки для вставки генерируются Python-блоком), флаги --type для переопределения типов колонок (классическая боль: ZIP-коды с ведущими нулями), метод drop_index(), чтение SQL из stdin и автоопределение первичного ключа для upsert. Главная фича — переключение SQLite strict-режима у существующих таблиц через механизм transform. The sqlite-utils saga continues (the "built by Fable" 4.0 release featured in our July 7 and 9 digests): a minor 4.1 that's interesting less for its features than for where they came from. Willison had Codex review the repo's open issues and pick "easy to implement" candidates — which is how the release got the --code option (rows generated by a Python block), --type flags to override column type detection (the classic ZIP-codes-with-leading-zeros pain), a drop_index() method, SQL from stdin, and automatic primary-key inference for upserts. The headline feature: toggling SQLite strict mode on existing tables via the transform mechanism.

Метод: два переносимых приёмаThe method: two transferable tricks

Первый — «агент-старатель» в трекере: вместо того чтобы самому перебирать бэклог, отдать агенту список открытых issues с задачей «найди то, что дёшево реализовать сейчас». Бэклог зрелого проекта — это готовая, годами валидированная очередь пожеланий; агент превращает её в конвейер маленьких побед. Второй — промпт ручной проверки: попросить модель «вручную поупражняться» с новой возможностью ( manually exercise the new .transform(strict=) option) вне автотестов — этот прогон вскрыл два бага, которые тесты пропустили. Приём прямо дополняет паттерн кросс-модельного ревью из истории 4.0. First — the "prospector agent" in the tracker: instead of combing the backlog yourself, hand the agent the open-issues list with the task "find what's cheap to implement now." A mature project's backlog is a ready-made, years-validated queue of wishes; the agent turns it into a conveyor of small wins. Second — the manual-exercise prompt: ask the model to "manually exercise" a new capability (manually exercise the new .transform(strict=) option) outside automated tests — that pass surfaced two bugs the tests had missed. It directly complements the cross-model review pattern from the 4.0 story.

Почему это важноWhy it matters

Оба приёма переносимы на любой проект с трекером: «агент-старатель» снимает главный барьер работы с бэклогом (никто не хочет его разбирать), а «ручная проверка промптом» — дешёвый слой верификации поверх тестов, особенно ценный для кода, который писал другой агент. Заметна и общая траектория Уиллисона: Fable пишет релиз, GPT ревьюит, Codex добывает фичи — мультивендорный конвейер в одном человеко-проекте. Both tricks transfer to any project with a tracker: the "prospector agent" removes the main backlog barrier (nobody wants to triage it), and the manual-exercise prompt is a cheap verification layer on top of tests — especially valuable for code another agent wrote. Note Willison's broader trajectory too: Fable writes the release, GPT reviews, Codex mines features — a multi-vendor pipeline in a one-person project.
