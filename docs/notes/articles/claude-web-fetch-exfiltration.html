Как исследователь заставил Claude слить твои самые личные данные How I tricked Claude into leaking your deepest, darkest secrets

Оригинал статьи ↗Original article ↗

Уиллисон разбирает исследование Аюша Пола «The Memory Heist»: proof-of-concept, показывающий, как заурядный на вид вопрос к claude.ai о ближайшей кофейне может обернуться незаметной утечкой личных данных пользователя на сторонний сервер. Ценность поста не в сенсационности находки, а в том, что он методично показывает, как две вполне полезные функции — память и веб-инструменты — в комбинации создают атаку, которую ни одна из них по отдельности не создаёт. Willison walks through researcher Ayush Paul's 'The Memory Heist': a proof-of-concept showing how an innocuous-looking question to claude.ai about a nearby coffee shop can turn into a silent leak of the user's personal data to an attacker-controlled server. The value of the post isn't the shock factor of the finding — it's how methodically it shows two genuinely useful features, memory and web tools, combining into an attack that neither one creates on its own.

Механика атаки: память плюс цепочка ссылокThe attack mechanics: memory plus a link chain

Claude.ai по умолчанию хранит память в виде ежедневных сводок разговоров и умеет искать по ним (conversation_search). Пол обнаружил, что инструмент web_fetch разрешал переходить по ссылкам, обнаруженным внутри уже загруженной страницы — то есть агент мог сам, без нового обращения пользователя, «провалиться» на следующую страницу, если та была указана в контенте первой. Он построил сайт-приманку — фейковую страницу кофейни, — где каждая следующая ссылка в цепочке кодировала в своём URL один символ извлекаемых данных. Попросив Claude «узнать что-нибудь про эту кофейню», Пол заставил модель объединить данные из памяти (имя, место работы и даже город, который Claude ни разу не сохранял явно, а вывел логически из названия хакатона в переписке) и передать их наружу по одному символу за раз, просто следуя по цепочке ссылок. Claude.ai stores memory by default as daily conversation summaries and can search them (conversation_search). Paul found that the web_fetch tool was allowed to follow links discovered inside a page it had already fetched — meaning the agent could 'fall through' to a next page on its own, without a fresh user request, if that page was referenced in the first one's content. He built a honeypot site — a fake coffee-shop page — where each successive link in a chain encoded one character of the data being exfiltrated in its URL. By asking Claude to 'find out something about this coffee shop,' Paul got the model to combine data from memory (name, employer, and even a hometown Claude had never explicitly stored but had inferred from a hackathon name mentioned in chat) and send it out one character at a time, simply by following the link chain.

Почему это не баг одной функции, а дыра на стыкеWhy this isn't a bug in one feature — it's a gap between them

Ключевой урок, который выделяет Уиллисон: ни память, ни web_fetch сами по себе не выглядят опасными, и обе прошли обычную проверку с точки зрения своей изолированной функциональности. Уязвимость возникла именно из-за композиции — память дала агенту данные, которые в принципе не должны покидать разговор, а свобода перехода по вложенным ссылкам дала канал, по которому эти данные можно вынести наружу без ведома пользователя. Это иллюстрация общего класса проблем агентных систем с инструментами: разрешения нужно оценивать не по каждому инструменту отдельно, а по тому, какие данные становятся доступны при их совместном использовании. The core lesson Willison highlights: neither memory nor web_fetch looks dangerous in isolation, and both passed ordinary scrutiny as standalone features. The vulnerability came specifically from composition — memory gave the agent data that should never leave the conversation, and the freedom to follow nested links gave it a channel to carry that data out without the user's knowledge. It's an illustration of a general class of problems in tool-using agent systems: permissions need to be evaluated not per tool, but by what data becomes reachable when tools are combined.

Как Anthropic закрыла дыруHow Anthropic closed the hole

Fix оказался узким и не потребовал отключать ни память, ни web_fetch целиком: Anthropic запретила инструменту переходить по ссылкам, обнаруженным во внешнем, уже загруженном контенте, оставив навигацию только по ссылкам из результатов поиска и введённым пользователем напрямую. Это сохраняет полезность обеих функций для нормального использования, но убирает именно тот канал, через который строилась цепочка эксфильтрации — модель больше не может самостоятельно «прогуляться» по произвольной последовательности ссылок, встроенных в контент сторонней страницы. The fix turned out to be narrow and didn't require disabling either memory or web_fetch entirely: Anthropic blocked the tool from following links discovered in external, already-fetched content, limiting navigation to search-result links and URLs the user provides directly. That preserves both features' usefulness for normal use while removing exactly the channel the exfiltration chain depended on — the model can no longer wander on its own through an arbitrary sequence of links embedded in a third-party page's content.
