DOOMQL: Doom, в котором каждый пиксель — это строка в SQLite DOOMQL: a Doom clone where every pixel is a row in SQLite

Оригинал статьи ↗Original article ↗

Уиллисон описывает проект Питера Гостева DOOMQL — и повод для заметки не столько сама игра, сколько намеренно неразумный вопрос, из которого она выросла: что, если SQLite — это не место, где игра хранит данные, а сам игровой движок? Гостев собрал DOOMQL с помощью GPT-5.6 Sol: получившаяся игра в стиле Doom работает на Python-скрипте для терминала, но движение, столкновения, врагов, бой, прогресс и вообще каждый RGB-пиксель на экране считает и хранит SQL. Уиллисон называет результат «сплошным весельем» — не потому что игра технически совершенна, а потому что сама постановка задачи заставляет модель придумывать нетривиальные способы выразить игровую логику через таблицы, представления и запросы. Willison writes up Peter Gostev's DOOMQL project — and the point of the post isn't really the game itself but the deliberately unreasonable question it grew from: what if SQLite isn't where a game stores its data, but the game engine itself? Gostev built DOOMQL with GPT-5.6 Sol: the resulting Doom-style game runs as a Python terminal script, but movement, collision, enemies, combat, progression, and literally every RGB pixel on screen are computed and stored by SQL. Willison calls the result "a lot of fun" — not because the game is technically polished, but because the premise itself forces the model to invent non-obvious ways to express game logic through tables, views, and queries.

Метод: неразумное ограничение как инструмент оценки моделиThe method: an unreasonable constraint as a model-evaluation tool

Это укладывается в паттерн, который Уиллисон регулярно освещает в своём блоге: ставить перед фронтир-моделью технически абсурдную, но чётко очерченную задачу — и смотреть, как далеко хватит её архитектурного мышления. SQL не проектировался как язык для расчёта коллизий или игрового цикла в реальном времени, поэтому «заставить» его исполнять эту роль требует от модели построить собственную непротиворечивую схему данных и логику обновления состояния, а не копировать шаблонное решение. Такой тест интереснее классических бенчмарков именно потому, что у него нет эталонного решения в обучающих данных — модель обязана рассуждать, а не подставлять запомненный паттерн. This fits a pattern Willison regularly covers on his blog: give a frontier model a technically absurd but sharply scoped task, then see how far its architectural reasoning stretches. SQL wasn't designed for real-time collision math or a game loop, so "forcing" it into that role requires the model to build its own consistent data schema and state-update logic rather than lean on a canned solution. That kind of test is more interesting than a standard benchmark precisely because there's no reference answer sitting in the training data — the model has to reason, not pattern-match a memorized template.

Как заглянуть внутрь: Datasette вместо чтения кодаLooking inside: Datasette instead of reading code

Уиллисон приводит инструкцию для запуска: git clone https://github.com/petergpt/doomql, затем uv run host/doomql.py — и это создаёт файл .doomql/doomql.sqlite прямо во время игры. Дальше он предлагает открыть эту базу через свой собственный инструмент Datasette и наблюдать за игровым состоянием не через код, а через таблицы: положение игрока, список активных врагов, очки здоровья — всё это обычные строки, которые можно просматривать и запрашивать SQL-запросами в реальном времени. Это частный случай более общего приёма, который Уиллисон продвигает: когда агент генерирует artefact с прозрачным внутренним состоянием (тут — SQLite-база), у человека появляется способ проверить и понять его работу без построчного чтения сгенерированного кода — сам формат данных становится отладочным интерфейсом. Willison includes the run instructions: git clone https://github.com/petergpt/doomql, then uv run host/doomql.py — which creates a .doomql/doomql.sqlite file while you play. He then suggests opening that database with his own Datasette tool and watching the game state not through code but through tables: player position, the list of active enemies, health points — all of it is just ordinary rows you can browse and query with SQL in real time. It's a specific instance of a broader technique Willison advocates: when an agent produces an artifact with transparent internal state (here, a SQLite database), a human gets a way to verify and understand its work without reading the generated code line by line — the data format itself becomes a debugging interface.
