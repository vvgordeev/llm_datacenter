Релиз: llm-meta-ai 0.1 Release: llm-meta-ai 0.1

Оригинал статьи ↗Original article ↗

Саймон Уиллисон получил всего несколько дней предварительного доступа к API новой модели Meta — Muse Spark 1.1 — и этого хватило, чтобы за то же время собрать llm-meta-ai: плагин для его собственного CLI-инструмента llm, дающий доступ к модели из командной строки и из Python-библиотеки. Это не разовый эксперимент, а рутинный для Уиллисона приём: у него уже собрана экосистема из десятков похожих плагинов под разных провайдеров, и появление новой модели означает не «искать, как к ней обращаться», а «написать тонкую обёртку по знакомому шаблону и подключить её к тому же интерфейсу, что и всё остальное». Simon Willison had only a few days of preview access to Meta's new model API — Muse Spark 1.1 — and that was long enough to put together llm-meta-ai: a plugin for his own llm CLI tool that gives command-line and Python-library access to the model. This isn't a one-off experiment; it's a routine move for Willison, who already maintains a whole ecosystem of similar plugins for different providers. A new model showing up doesn't mean "figure out how to call it" — it means "write a thin wrapper to the familiar template and plug it into the same interface as everything else."

Один интерфейс поверх множества провайдеровOne interface over many providers

Ключевая идея подхода Уиллисона — модели разных вендоров живут в llm под общим пространством имён с префиксом провайдера: новая модель доступна как meta-ai/muse-spark-1.1, и вызов выглядит одинаково для любого провайдера — llm -m meta-ai/muse-spark-1.1 "What is the capital of France?". Установка занимает три команды: поставить сам llm (uv tool install llm), поставить плагин (llm install llm-meta-ai) и сохранить ключ (llm keys set meta-ai). Для пользователя это означает, что переключение между Claude, GPT, Gemini или новой моделью Meta — вопрос смены одного аргумента -m, а не изучения нового SDK и другого набора соглашений по каждому провайдеру. The core idea behind Willison's approach is that different vendors' models live inside llm under a shared provider-prefixed namespace: the new model is available as meta-ai/muse-spark-1.1, and the call looks identical regardless of provider — llm -m meta-ai/muse-spark-1.1 "What is the capital of France?". Setup takes three commands: install llm itself (uv tool install llm), install the plugin (llm install llm-meta-ai), and store the key (llm keys set meta-ai). For the user, that means switching between Claude, GPT, Gemini, or Meta's new model is a matter of changing one -m argument, not learning a new SDK and a different set of per-provider conventions.

Рассуждающая модель и картинки — тоже через общий контрактA reasoning model and images — through the same contract

Muse Spark 1.1 — рассуждающая модель: она «думает» перед ответом, и токены рассуждения включаются в бюджет выходных токенов. Плагин добавляет управление глубиной рассуждений через параметр с уровнями none, minimal, low, medium, high и xhigh — то есть эта настройка вписана в тот же унифицированный интерфейс, что и у других рассуждающих моделей в экосистеме llm, а не изобретена заново под конкретного вендора. Отдельно плагин поддерживает передачу изображений по URL — модель может описывать картинки из интернета прямо по ссылке, без предварительного скачивания и кодирования в base64 вручную. Muse Spark 1.1 is a reasoning model: it "thinks" before answering, and reasoning tokens count against the output token budget. The plugin exposes control over reasoning depth through a parameter with levels none, minimal, low, medium, high, and xhigh — meaning this setting slots into the same unified interface already used for other reasoning models in the llm ecosystem, rather than being reinvented per vendor. Separately, the plugin supports passing images by URL — the model can describe pictures straight from a web link, with no manual downloading or base64 encoding needed first.

Почему это интересно как подходWhy this matters as an approach

Показательная деталь: тестируя плагин, Уиллисон нашёл и тут же исправил баг в самом ядре llm 0.31.1 — некоторые провайдеры возвращали ошибку JSON на вызов инструмента с пустыми аргументами через эндпоинт OpenAI Chat Completions. То есть работа над плагином для одной новой модели улучшила инфраструктуру для всех остальных. Это и есть суть паттерна: вместо того чтобы каждый раз оценивать новую модель через отдельный, специфичный для вендора клиент, у практика есть один устойчивый инструмент, который растёт вширь с каждым новым релизом — а любой найденный по пути баг чинится один раз в общем месте, а не в N местах. A telling detail: while testing the plugin, Willison found and immediately fixed a bug in llm 0.31.1's core — some providers returned a JSON error on a tool call with empty arguments through the OpenAI Chat Completions endpoint. So the work on a plugin for one new model improved the infrastructure for everything else. That's the essence of the pattern: instead of evaluating every new model through a separate, vendor-specific client, a practitioner keeps one durable tool that grows wider with each new release — and any bug found along the way gets fixed once in a shared place, not N times.
