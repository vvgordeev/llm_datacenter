GPT-5.6 больше не вызывает инструменты по одному — теперь он пишет код, сократив токены на 63,5% GPT-5.6 stopped calling tools one at a time — it writes code now, and cut tokens 63.5%

Оригинал статьи ↗Original article ↗

Автор начинает с прямого утверждения: главное в релизе GPT-5.6 — не сама модель, а тихо переписанный способ вызова инструментов, о котором почти никто не говорит. Речь о Programmatic Tool Calling — новом режиме в Responses API, который меняет саму механику того, как агент работает с внешними инструментами на многошаговых задачах. The author opens with a blunt claim: the real substance of the GPT-5.6 release isn't the model itself, but a quietly rewritten way of calling tools that almost nobody is talking about. That's Programmatic Tool Calling — a new mode in the Responses API that changes the actual mechanics of how an agent works with external tools on multi-step tasks.

Старый цикл vs код-оркестраторThe old loop vs. code-as-orchestrator

В привычной схеме модель делает один шаг за раз: формирует структурированный запрос на вызов инструмента, ваш код его выполняет, весь результат целиком возвращается обратно в контекст модели, и только после этого модель решает, что делать дальше. На задачах с большим числом вызовов инструментов это дорого — каждый промежуточный результат целиком проходит через контекст. Programmatic Tool Calling меняет расстановку ролей: теперь GPT-5.6 сам пишет короткую программу на JavaScript, которая вызывает нужные инструменты, передаёт результаты между вызовами и обрабатывает промежуточные данные — а модель получает уже свёрнутый итог, а не сырой поток ответов на каждом шаге. In the familiar setup, the model moves one step at a time: it emits a structured request to call a tool, your code runs it, the entire result comes back into the model's context, and only then does the model decide on the next move. On tasks with many tool calls this is expensive — every intermediate result passes through the context in full. Programmatic Tool Calling flips the roles: GPT-5.6 now writes a short JavaScript program itself that calls the needed tools, passes results between calls, and processes intermediate data — and the model gets back one collapsed result instead of a raw stream of responses at every step.

Изолированная среда V8 — без сетиAn isolated V8 runtime — no network access

Написанный моделью код исполняется в изолированной среде V8 (движок JavaScript из Chrome/Node.js), у которой нет доступа к сети — это ограничивает риск того, что модель через собственный код сделает что-то за пределами разрешённых инструментов. Автор подчёркивает, что режим подходит именно для «ограниченных» тул-хэви задач, где не требуется свежее суждение модели между каждым шагом: код может фильтровать большие объёмы промежуточных данных, оставлять только нужное и адаптировать порядок действий по ходу выполнения — без обращения к модели на каждой мелочи. The model-written code runs inside an isolated V8 runtime (the JavaScript engine behind Chrome/Node.js) with no network access — limiting the risk that the model's own code does something outside the allowed tools. The author stresses this mode fits "bounded," tool-heavy workflows that don't need fresh model judgment between every step: the code can filter large volumes of intermediate data, keep only what matters, and adapt the workflow along the way — without pinging the model for every small decision.

Цифры и когда это применятьThe numbers, and when to reach for it

Ключевая цифра статьи: один из ранних клиентов OpenAI запустил того же агента на тех же инструментах и получил сокращение токенов на 63,5% при той же задаче — вместе с более быстрым и более дешёвым выполнением по другим замерам (24% меньше выходных токенов и на 28% быстрее в одном из кейсов). Автор предупреждает: это не универсальная замена обычному tool calling — там, где агенту действительно нужно принимать решение после каждого шага на основе непредсказуемого результата, старая схема остаётся уместнее; выигрыш максимален именно на длинных, предсказуемых по структуре цепочках вызовов. The article's headline number: one of OpenAI's early customers ran the same agent on the same tools and saw a 63.5% cut in tokens on the identical task — alongside faster, cheaper execution on other measures (24% fewer output tokens and 28% faster in one case). The author cautions this isn't a universal replacement for ordinary tool calling — where an agent genuinely needs to decide after every step based on an unpredictable result, the old scheme still fits better; the payoff is biggest specifically on long, structurally predictable chains of tool calls.
