Rewriting Bun in Rust: миллион строк за 11 дней силами агентов Rewriting Bun in Rust: a million lines in 11 days, by agents

Оригинал статьи ↗Original article ↗

Создатель JavaScript-рантайма Bun Джарред Самнер описал, как один инженер с Claude Code перенёс 535 тысяч строк кода с Zig на Rust за 11 дней: 1448 файлов, 6502 коммита, итоговый диф больше миллиона строк и примерно $165 000 в API-токенах (5.9 млрд некешированных входных, 690 млн выходных). Это, вероятно, самый детальный публичный разбор крупной агентной миграции на сегодня — с методом, цифрами и честным списком того, что шло не так. Bun creator Jarred Sumner describes how one engineer with Claude Code ported 535K lines from Zig to Rust in 11 days: 1,448 files, 6,502 commits, a final diff of over a million lines and roughly $165,000 in API tokens (5.9B uncached input, 690M output). It is probably the most detailed public account of a large agentic migration to date — with the method, the numbers, and an honest list of what went wrong.

Метод: тестовый набор как «оракул соответствия»The method: the test suite as a conformance oracle

Ключевое условие, сделавшее проект возможным: тесты Bun написаны на TypeScript и не зависят от языка реализации — значит, они годятся как conformance suite, внешний судья правильности. Правило «ноль тестов пропущено или удалено» превращает размытую задачу «перепиши на Rust» в измеримую: 60 624 теста и 1.39 млн assert'ов должны пройти. Подготовка заняла считанные часы: Самнер обсудил с Claude паттерны миграции и зафиксировал их в PORTING.md (какие конструкции Zig во что превращать), а отдельный workflow прошёл по всем структурам, проследил владение памятью и сгенерировал LIFETIMES.tsv с предложенными аннотациями времён жизни — оба документа затем прошли адверсариальное ревью. The key enabler: Bun's tests are written in TypeScript and don't depend on the implementation language — so they work as a conformance suite, an external oracle of correctness. The rule "zero tests skipped or deleted" turns the fuzzy task "rewrite it in Rust" into a measurable one: 60,624 tests and 1.39M assertions must pass. Preparation took hours, not weeks: Sumner discussed migration patterns with Claude and captured them in PORTING.md (which Zig constructs map to what), while a separate workflow walked every struct, traced memory ownership and generated LIFETIMES.tsv with proposed lifetime annotations — both documents then went through adversarial review.

Конвейер агентов: реализатор → два оппонента → фиксерThe agent pipeline: implementer → two adversaries → fixer

Каждый файл шёл по конвейеру из четырёх ролей: реализатор (видит исходный Zig, план и своё рассуждение) → два независимых адверсариальных ревьюера (видят только диф, задача — найти ошибки) → фиксер (применяет замечания). На пике одновременно работали ~64 агента в четырёх git-worktree, выдавая до 1300 строк в минуту — и каждая строка проходила через двух оппонентов до коммита. Ревьюеры ловили именно то, что страшно в таких портах: use-after-free при закрытии pipe (Box::leak вместо преждевременного drop), неверную нормализацию отрицательных timespec (floor vs trunc), потерю ленивости вычислений (unwrap_or vs unwrap_or_else). Every file went through a four-role pipeline: an implementer (sees the original Zig, the plan, and its own reasoning) → two independent adversarial reviewers (see only the diff, tasked to find bugs) → a fixer (applies the feedback). At peak, ~64 agents ran concurrently across four git worktrees, producing up to 1,300 lines per minute — and every line passed two adversaries before commit. The reviewers caught exactly what's scary in such ports: a use-after-free on pipe close (Box::leak instead of a premature drop), wrong normalization of negative timespec (floor vs trunc), and lost laziness (unwrap_or vs unwrap_or_else).

Что пошло не так — и чем закончилосьWhat went wrong — and how it ended

Список факапов не менее ценен, чем метод: агенты устраивали хаос с git stash/git reset, мешая друг другу; начинали «заглушать» функции вместо реального решения; забытый grep замораживал систему, а диск дважды переполнялся. Ответом стали жёсткие правила: запрет git-операций без коммита, запрет «медленных команд», изоляция рабочих деревьев. Семантические ловушки Zig→Rust (ленивость debug_assert!, bounds checks в release, нечётные срезы UTF-16, comptime-форматирование) отлавливались тем же адверсариальным ревью. Итог: 99.8% тестов на Linux x64 через 6 дней, все 6 платформ зелёные к 14-му дню проекта, 19 регрессий (исправлены), бинарник на 20% меньше, производительность +2–5%. Вывод Самнера: «один инженер сегодня может сделать намного больше, чем год назад». The failure list is as valuable as the method: agents wreaked havoc with git stash/git reset, stepping on each other; started stubbing functions instead of solving problems; a forgotten grep froze the machine, and the disk filled up twice. The answer was hard rules: no git operations without committing, a ban on "slow commands," isolated worktrees. Zig→Rust semantic traps (lazy debug_assert!, release-build bounds checks, odd UTF-16 slices, comptime format strings) were caught by the same adversarial review. The outcome: 99.8% of tests passing on Linux x64 in 6 days, all six platforms green by day 11, 19 regressions (all fixed), a 20% smaller binary, +2–5% performance. Sumner's takeaway: "one engineer can do far more today than a year ago."

Почему это важноWhy it matters

Рецепт переносим: если у системы есть независимый от реализации набор тестов, он превращается в оракул, против которого агенты могут мигрировать код почти автономно — а «реализатор + два оппонента + фиксер» страхует от плазибельно-неверного кода. Это меняет экономику больших миграций (legacy-стеки, смена языков, обновления фреймворков): $165K и 11 дней против человеко-лет ручного порта. The recipe transfers: if a system has an implementation-independent test suite, it becomes an oracle against which agents can migrate code near-autonomously — and "implementer + two adversaries + fixer" insures against plausible-but-wrong code. That changes the economics of large migrations (legacy stacks, language switches, framework upgrades): $165K and 11 days versus person-years of manual porting.
