Дайте Fable 5 задание, а не промпт Give Fable 5 a mission, not just a "prompt"

Оригинал статьи ↗Original article ↗

Центральный тезис автора: для моделей уровня Fable 5, рассчитанных на долгую самостоятельную работу, короткий промпт — это потраченный впустую потенциал модели. Автор проводит границу между «промптом» в старом смысле (короткой инструкцией на один ответ) и «мишн-брифом» (mission brief) — операционным заданием с полным контуром: определить цель → передать контекст → задать границы допустимого → сформулировать критерий успеха → потребовать доказательства прогресса → проверить результат. Разница не косметическая: тонкий промпт оставляет модели слишком много неявных решений, которые она либо додумывает произвольно, либо решает слишком консервативно — а щедро описанный бриф даёт ей пространство работать на всю мощность, не отклоняясь от намерения автора. The author's central claim: for models like Fable 5, built for long, unsupervised runs, a short prompt is wasted model potential. The piece draws a line between a "prompt" in the old sense (a short one-shot instruction) and a "mission brief" — an operating assignment with a full loop: define the objective → supply context → set boundaries → specify success criteria → require evidence of progress → review the output. The difference isn't cosmetic: a thin prompt leaves the model too many implicit decisions, which it either fills in arbitrarily or resolves too conservatively — a generously specified brief gives it room to work at full capacity without drifting from the author's intent.

Анатомия цикла: шесть частей, а не одна строкаAnatomy of the loop: six parts, not one line

Автор раскладывает практический шаблон долгоживущего агентного цикла на шесть элементов. Триггер запускает итерацию — это может быть расписание cron, упавшая CI-проверка, вебхук или самостоятельный ритм модели через /loop. Правила подгружают замороженные знания о проекте и накопленную память между итерациями. Исполнитель делает ровно одну ограниченную единицу работы — не расплывчатое «продвинься дальше», а конкретное «почини следующий падающий тест» или «обработай следующие десять файлов»: чем у́же и измеримее единица работы, тем меньше шансов, что модель наберёт незамеченных ошибок за один проход. Замыкают цикл верификатор, память и правила остановки. The author lays out a practical template for a long-running agent loop as six pieces. A trigger kicks off an iteration — a cron schedule, a failed CI check, a webhook, or the model's own pacing via /loop. Rules load frozen project knowledge plus accumulated memory between iterations. The executor does exactly one bounded unit of work — not a vague "make progress" but a concrete "fix the next failing test" or "process the next ten files": the narrower and more measurable the unit, the less chance the model racks up unnoticed mistakes in a single pass. The loop closes with a verifier, memory, and stop rules.

Верификатор в свежем контексте: почему самокритика слабееA fresh-context verifier: why self-critique is weaker

Отдельное и самое конкретное место статьи — про верификатора. Автор настаивает, что проверка должна идти в отдельном, свежем контексте: тот, кто выполнял работу, видит собственную цепочку рассуждений и потому склонен себя оправдывать, а верификатор должен видеть только готовый артефакт и критерий (рубрику) — без доступа к тому, как агент пришёл к результату. В подтверждение автор ссылается на промптинг-гайд Anthropic по Fable 5: независимые верификаторы-субагенты в свежем контексте поймали около 73% специально внесённых ошибок, тогда как самокритика (когда модель проверяет саму себя в том же контексте) — всего 7–33%. Практический вывод для читателя: если агентный цикл уже настроен, но качество плавает, первое, что стоит проверить — не делит ли верификатор контекст с исполнителем, и достаточно ли ограничена единица работы для одной итерации. The piece's single most concrete point is about the verifier. The author insists verification must run in a separate, fresh context: whoever did the work sees its own reasoning trail and is therefore inclined to excuse itself, while the verifier should see only the finished artifact and the rubric — with no access to how the agent got there. In support, the author cites Anthropic's Fable 5 prompting guide: independent fresh-context verifier subagents caught around 73% of seeded issues, versus just 7–33% for self-critique (the model checking itself within the same context). The practical takeaway for the reader: if an agent loop is already set up but quality is inconsistent, the first thing to check is whether the verifier shares context with the executor, and whether the unit of work is bounded enough for a single iteration.
