# Требования к продукту — журнал инкрементов

Требования ведутся как **append-only журнал**: каждый файл — неизменяемый инкремент (запрос на изменение продукта). Существующие инкременты не редактируются задним числом — новые решения оформляются новым инкрементом.

## Принцип разделения с остальной документацией

- **Требования (эта папка)** — история *намерений*: что и почему решили изменить. После реализации файл остаётся как исторический документ.
- **`docs/features/*` и `docs/architecture.md`** — *актуальное состояние* системы. Обновляются в том же коммите, что и код. На вопрос «как сейчас работает X» отвечают они, а не требования.
- **`docs/roadmap.md`** — что планируется, но ещё не оформлено в требования.

## Статусы инкремента (в шапке файла)

| Статус | Значение |
|---|---|
| draft | черновик, обсуждается |
| accepted | согласован, содержит решения по открытым вопросам и ТЗ; можно реализовывать |
| implemented | реализован; актуальное поведение смотреть в docs/features/* |

## Сводный документ требований

Постоянно поддерживаемого сводного файла НЕТ — намеренно: сводное непротиворечивое описание текущего состояния — это `docs/features/*` (обновляются вместе с кодом). Второй рукописный «источник истины» неизбежно начал бы противоречить первому. Когда нужен сводный документ для внешнего потребителя — он **генерируется по запросу** (слияние инкрементов с разрешением перекрытий в пользу реализованного, файл `consolidated-YYYY-MM.md` с пометкой «производный снапшот, источник истины — docs/features/*»).

## Формат нового инкремента

Файл `vX.Y-краткое-имя.md`: статус и дата → исходные требования → выявленные конфликты с текущей системой → принятые решения по открытым вопросам → допущения → техническое задание (модель данных, сервисы, UI, тесты, доки) → фазы реализации.
