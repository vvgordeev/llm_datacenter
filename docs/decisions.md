# Архитектурные решения (ADR)

Краткие записи «почему так». Формат: контекст → решение → последствия. **Новые решения дописывать сверху.**

## ADR-6: Дизайн-система Neural Radar
Контекст: исходный дизайн дашборда выглядел устаревшим; пользователь предоставил пример (design_example/: токены + макет) и утвердил прототип.
Решение: дашборд переведён на токены Neural Radar (Orbitron/Rajdhani/Inter/JetBrains Mono, циан+фиолет, glow); контент разнесён по вкладкам «Радар» и «Эксперты», срезы День/Неделя/Месяц — в шапке. Страницы конспектов перекрашены через алиасы старых CSS-переменных в style.css — их HTML не менялся.
Последствия: у видео-конспектов появилось поле `cover` в данных; шрифты грузятся с Google Fonts (внешняя зависимость, для GitHub Pages приемлемо); design_example/ хранится как эталон токенов.

## ADR-5: Конспекты видео — через Gemini API, кадры — yt-dlp/ffmpeg
Контекст: нужны тезисы, таймкоды и кадры значимых демонстраций из YouTube-видео; Claude не анализирует видеоряд.
Решение: Gemini API принимает YouTube URL напрямую и возвращает конспект + таймкоды «значимых демонстраций»; кадры извлекаются локально yt-dlp+ffmpeg только по этим таймкодам (≤6, jpg).
Последствия: зависимость от GEMINI_API_KEY и лимитов free tier (лимит видео/день в sources.yaml); fallback — конспект без кадров.

## ADR-4: Недельные/месячные срезы считаются на клиенте
Контекст: нужны приоритизированные срезы за неделю/месяц без пересборки архива.
Решение: дневные JSON — единственный источник; клиент группирует по cluster_id, берёт свежайшую версию сюжета, сортирует по impact_score, показывает топ-N.
Последствия: нет отдельных агрегатных файлов и рассинхрона; за месяц грузится ~30 небольших JSON — приемлемо, при росте можно добавить прекомпиляцию.

## ADR-3: Контент хранится двуязычно (RU+EN), а не переводится на лету
Контекст: на дашборде нужен переключатель языка; сайт статический, переводить на клиенте нечем.
Решение: пайплайн генерирует все текстовые поля парой {ru, en}; UI-строки — словарь i18n в app.js.
Последствия: данные ~×2 по тексту и чуть дороже генерация; зато переключение мгновенное и офлайн.

## ADR-2: Ежедневный запуск — облачная routine Claude Code
Контекст: сбор в 8:00 GMT+3 не должен зависеть от включённого ПК пользователя.
Решение: scheduled cloud agent (cron 0 5 * * * UTC) клонирует репо, выполняет /digest, пушит; тот же скилл запускается локально для отладки.
Последствия: репозиторий должен жить на GitHub; секреты (GEMINI_API_KEY) настраиваются в облачном окружении routine.

## ADR-1: Статический сайт на GitHub Pages вместо приложения с бэкендом
Контекст: нужен ежедневно обновляемый дашборд с архивом, доступный с любого устройства, без затрат на инфраструктуру.
Решение: репо = хранилище (JSON по дням) + сайт (docs/, vanilla JS без сборки); публикация — git push.
Последствия: нет БД и серверного кода; архив версионируется git'ом; ограничение — вся логика срезов на клиенте (см. ADR-4).

## ADR-0: Формат этого журнала
Контекст: нужен лёгкий способ фиксировать крупные/неочевидные решения, чтобы они не терялись и не переоткрывались.
Решение: каждое значимое архитектурное решение — короткая запись здесь (3–6 строк): контекст, что решили, последствия/цена. Новые — сверху. Мелкие детали реализации сюда не пишем — им место в `docs/features/*`.
Последствия: история решений живёт рядом с кодом; ревью новых ADR — часть обычного PR.
