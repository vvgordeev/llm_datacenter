# Метод работы над проектом (конвенции)

Канон подхода. Обновляется централизованно в репозитории практик и подтягивается скиллом `/sync-practices`. Проектная специфика — в `AGENTS.md` и `docs/*`.

## 1. Документация — источник истины, и она правится вместе с кодом
- **`docs/features/*.md`** — *актуальное состояние* системы по областям. Отвечает на «как сейчас работает X». Обновляется **в том же коммите**, что и изменение поведения (правило №1). Устаревший док хуже отсутствующего.
- **`docs/architecture.md`** — слои, **инварианты** (что нельзя ломать), матрица доступа. Читается **первым** перед любой задачей.
- **`docs/decisions.md`** — ADR (Architecture Decision Records): контекст → решение → последствия. Новые записи **сверху**. Заводится на каждое крупное/неочевидное решение.
- **`docs/requirements/`** — **append-only журнал инкрементов** (запросы на изменение продукта). Не редактируется задним числом; новое решение = новый инкремент. См. `docs/requirements/README.md`.
- **`docs/roadmap.md`** — что планируется, но ещё не оформлено в требования.

Разделение: требования — история *намерений*; features/architecture — *текущее состояние*. Сводный документ не поддерживается вручную — генерируется по запросу из инкрементов с пометкой «производный снапшот».

## 2. Инварианты священны
Ключевые правила системы (неизменяемость опубликованного, ссылки-не-копии, серверная авторитетность и т.п.) живут в `docs/architecture.md` и не нарушаются. Их нельзя менять «мимоходом» — только осознанным решением с ADR.

## 3. Тесты и CI
- Split: **unit** (чистая логика, без БД) и **integration** (против реальной dev-БД, последовательно, с уникальными фикстурами и cleanup по внешним ключам).
- Обязательный минимум перед коммитом: lint + typecheck + unit. Integration — если менялись сервисы/схема.
- CI повторяет этот минимум на каждый push + PR.

## 4. Делегирование субагентам — только через `/delegate`
Дорогая модель-оркестратор; дешёвые субагенты — на однозначную рутину по паттернам. Протокол: декомпозиция → назначение → оценка выгоды в токенах → **подтверждение пользователя** → исполнение → приёмка (diff + проверки). Никогда не делегируются: auth/безопасность, инварианты architecture.md, миграции БД, отладка неясных багов.

## 5. Трекинг задач и багов — во внешнем трекере
Не в эфемерной памяти агента. Крупные фичи — issue со связью на код. Метки-области + типовые (Bug/Feature/Improvement). Для автономных агентов — API key трекера.

## 6. Прочее
- Строки UI — через i18n, не хардкод.
- Секреты — только в переменных окружения/панелях, не в коде, не в чате, не в git.
- Инфраструктура портативна: один Docker-образ, миграции идемпотентны, деплой воспроизводим.
