# /digest — ежедневный пайплайн сводки

Собирает сводку за дату `TARGET_DATE` (по умолчанию — сегодня по GMT+3). Перед работой прочитай `docs/architecture.md` (инварианты!) и `config/sources.yaml`.

## Шаг 0. Подготовка

1. Прочитай `config/sources.yaml` — единственный источник того, ЧТО собирать. Модели с `enabled: false` пропускаются.
2. Прочитай `data/state.json` (если нет — создай `{"seen_urls": {}, "clusters": {}}`): `seen_urls` — URL → дата первой встречи; `clusters` — cluster_id → {последняя дата, краткое описание сюжета}. Записи старше 60 дней вычищай.
3. Если `data/daily/<TARGET_DATE>.json` уже существует — это **пересбор**: перезапиши его, но не трогай другие дни (инвариант 1). ВАЖНО: при пересборе URL, впервые увиденные в сам `TARGET_DATE` (дата в `seen_urls` равна целевой), дублями НЕ считаются — иначе пересбор выкинет всё, что собрал утренний прогон. Дубль — только то, что видели ДО целевой даты.
4. Инструменты для видео: если `yt-dlp`/`ffmpeg` недоступны — `python -m pip install --quiet yt-dlp imageio-ffmpeg` (extract_frames.py умеет использовать pip-версии; в облачном окружении их нет — ставить при каждом прогоне).

## Шаг 1. Сбор новостей

Для каждой включённой модели из `models` + общий поиск:
- WebFetch официального блога; WebSearch «<модель> announcement/release/feature» за окно `limits.lookback_hours`.
- WebSearch по community-источникам: топовые обсуждения на Hacker News, указанных сабреддитах.
- Кандидат — событие о **новой функции, релизе или новом подходе к использованию LLM**. Не бери: слухи без первоисточника, маркетинг без содержания, финансовые новости без продуктовой части.

Дедупликация: URL уже в `seen_urls` → пропусти. Событие продолжает известный сюжет из `clusters` → используй существующий `cluster_id`, иначе заведи новый (kebab-case: `<компания>-<суть>`).

Отбери максимум `limits.max_news_items` самых значимых.

## Шаг 2. Обогащение каждой новости

Для каждой новости собери (WebSearch/WebFetch):
- **summary** — 2–3 предложения: что именно появилось и что это даёт.
- **opinions** — 2–4 реакции с РАЗНЫМИ позициями (positive / skeptical / neutral) от сообщества и экспертов, каждая с точным URL (тред HN, пост Reddit, статья). Указывай автора/площадку. Не выдумывай цитаты — только реально найденное; если реакций ещё нет, оставь пустой список.
- **use_cases** — 1–3 конкретных кейса применения со ссылками (из обсуждений, блогов, документации).

## Шаг 3. Ранжирование — impact_score (главное в продукте)

Оцени каждую новость по рубрике, выставь **impact_score 1–10** и одну строку **impact_reason**:

| Балл | Критерий |
|---|---|
| 9–10 (paradigm) | Меняет саму парадигму использования LLM; через год так будут работать все |
| 7–8 (major) | Заметно меняет существующие подходы ИЛИ открывает новый класс сценариев |
| 4–6 (notable) | Полезное развитие в рамках текущих подходов; влияет на выбор инструментов |
| 1–3 (minor) | Инкрементальное улучшение, подходы не меняет |

Взвешивай: (a) меняет ли текущие подходы, (b) открывает ли новые, (c) широта применимости (все пользователи LLM или ниша), (d) сила реакции сообщества. Оценивай консервативно: 9–10 — редкость. Для продолжения сюжета оценка может расти/падать относительно вчерашней — срезы возьмут максимум по кластеру.

## Шаг 4. Экспертный контент

По списку `experts` из конфига + WebSearch «best/popular LLM article/video this week»:
- Отбери до `limits.max_articles_per_day` статей и `limits.max_videos_per_day` видео, опубликованных недавно и вызвавших резонанс, о ПОДХОДАХ к использованию LLM (не новости-пересказы).
- **Статья:** WebFetch полного текста → страница `docs/notes/articles/<slug>.html`: ключевые тезисы (нумерованный список), схемы — восстанови как inline-SVG с пояснением (образец: `docs/notes/articles/demo-article.html`; используй `../note.css`, `../note.js`, двуязычные `<span data-lang="ru|en">`).
- **Видео:** `python scripts/gemini_video.py <url> --out /tmp/summary.json` → страница `docs/notes/videos/<slug>.html` с тезисами и таймкодами-ссылками (`https://youtube.com/watch?v=ID&t=<sec>`); кадры: `python scripts/extract_frames.py <url> --from-summary /tmp/summary.json --out-dir docs/media/frames/<slug>` → `<img>` вместо плейсхолдеров. Если GEMINI_API_KEY или yt-dlp/ffmpeg недоступны — конспект по субтитрам/описанию без кадров, пометь это на странице.
- **Если YouTube блокирован сетью окружения** (WebFetch/yt-dlp не достают youtube.com — так в облачной routine): НЕ отбрасывай видео. Gemini API получает ролик на стороне серверов Google — `gemini_video.py` работает и за блокировкой. Верификация автора/содержания — по самому конспекту от Gemini (он видит видео) плюс косвенно: ссылка на видео со страницы/поста автора, найденного WebSearch. Кадры в этом случае пропусти, на странице конспекта поставь пометку «кадры недоступны в окружении сбора».
- Slug — `scripts/common.py: slugify(en-заголовок)`.

## Шаг 5. Запись данных

Собери `data/daily/<TARGET_DATE>.json` строго по схеме (все текстовые поля — `{"ru": ..., "en": ...}`, инвариант 2):

```json
{
  "date": "YYYY-MM-DD",
  "generated_at": "ISO-8601 UTC",
  "news": [{
    "id": "<date>-<slug>", "cluster_id": "<сюжет>", "models": ["<tag из sources.yaml>"],
    "title": {"ru", "en"}, "summary": {"ru", "en"},
    "impact_score": 1-10, "impact_reason": {"ru", "en"},
    "source_url": "...",
    "opinions": [{"author", "stance": "positive|skeptical|neutral", "quote": {"ru", "en"}, "url"}],
    "use_cases": [{"text": {"ru", "en"}, "url"}]
  }],
  "expert_content": [{
    "id", "type": "article|video", "author", "title": {"ru", "en"}, "url",
    "summary": {"ru", "en"}, "note_page": "notes/<...>.html"
  }]
}
```

Без поля `demo`. Обнови `data/state.json` (новые URL и кластеры). Демо-файлы (`demo: true`) при первом реальном прогоне удали из `data/daily/` и `docs/data/daily/`, демо-страницы `demo-article.html`/`demo-video.html` оставь — на них ссылается документация.

## Шаг 6. Билд и проверки

```
python scripts/build_site.py        # index.json + копия в docs/data/
python -m pytest tests/unit -q     # должны проходить
```
Валидность JSON проверь чтением; убедись, что у каждой новости есть impact_score, source_url и оба языка.

## Шаг 7. Публикация

Коммит: `digest: сводка за <TARGET_DATE>` (+ Co-Authored-By по конвенции). В облачной routine — `git push`; локально — спроси пользователя, пушить ли. При пересборе коммить обновлённый файл так же; если после всех шагов коммитить действительно нечего — не выходи молча, а явно объясни в финальном отчёте, почему (что искал, что нашёл, почему отфильтровал).

## Ограничения

- Не выдумывай новости, цитаты и ссылки. Нет уверенности в реальности события — не включай.
- Мало событий за день — это нормально: лучше 3 настоящих новости, чем 12 натянутых.
- Пиши двуязычный контент сразу, не «переведу потом».
