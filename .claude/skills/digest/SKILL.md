# /digest — ежедневный пайплайн сводки

Собирает сводку за дату `TARGET_DATE` (по умолчанию — сегодня по GMT+3). Перед работой прочитай `docs/architecture.md` (инварианты!) и `config/sources.yaml`.

## Шаг 0. Подготовка

1. Прочитай `config/sources.yaml` — единственный источник того, ЧТО собирать. Модели с `enabled: false` пропускаются.
2. Прочитай `data/state.json` (если нет — создай `{"seen_urls": {}, "clusters": {}}`): `seen_urls` — URL → дата публикации; `clusters` — cluster_id → {последняя дата, краткое описание сюжета}. Записи старше 60 дней вычищай. ПРАВИЛО ЗАПИСИ: в `seen_urls` попадают ТОЛЬКО `source_url` опубликованных новостей и URL опубликованных конспектов. Ссылки мнений/кейсов и отклонённые кандидаты НЕ записываются — завтра их дёшево переоценить заново; иначе фильтр с каждым днём душит сбор.
3. Если `data/daily/<TARGET_DATE>.json` уже существует — это **пересбор**: перезапиши его, но не трогай другие дни (инвариант 1). ВАЖНО: при пересборе URL, впервые увиденные в сам `TARGET_DATE` (дата в `seen_urls` равна целевой), дублями НЕ считаются — иначе пересбор выкинет всё, что собрал утренний прогон. Дубль — только то, что видели ДО целевой даты.
4. Инструменты для видео: если `yt-dlp`/`ffmpeg` недоступны — `python -m pip install --quiet yt-dlp imageio-ffmpeg` (extract_frames.py умеет использовать pip-версии; в облачном окружении их нет — ставить при каждом прогоне).

## Шаг 1. Сбор новостей

Обязательная широта прохода (фиксируй результат каждого источника для run_log, шаг 6.5).

СЕТЕВАЯ РЕАЛЬНОСТЬ: в облачной песочнице egress-прокси отдаёт 403 на БОЛЬШИНСТВО прямых fetch — блоги вендоров, агрегаторы, hn.algolia.com (подтверждено run_log 2026-07-09). Прямой fetch источника пробуй ОДИН раз (для статуса в run_log), не ретраить; **основной канал discovery в облаке — WebSearch**, он работает всегда. В локальных прогонах прямые fetch обычно доступны — используй их.

- **Каждая** включённая модель из `models`: одна попытка WebFetch блога + МИНИМУМ 2 WebSearch-запроса («<модель> announcement <месяц год>», «<модель> new feature/release») за окно `limits.lookback_hours`.
- **Агрегаторы** из секции `aggregators`: одна попытка WebFetch за целевую дату; при 403 — WebSearch по содержимому агрегатора («site:<домен> <дата>» или «AI news digest <дата>»). Из агрегаторов берутся события и ПЕРВИЧНЫЕ ссылки (сам агрегатор в сводку не ссылается, кроме случаев, когда он и есть первоисточник мнения).
- **Hacker News**: попытка API `https://hn.algolia.com/api/v1/search_by_date?tags=story&query=<LLM|модель>&numericFilters=created_at_i><unix-24h>`; при 403 — WebSearch «site:news.ycombinator.com <тема>» и «Hacker News <тема> discussion».
- Остальные community-источники из конфига (Reddit, X) — через WebSearch.
- Кандидат — событие о **новой функции, релизе или новом подходе к использованию LLM**. Не бери: слухи без первоисточника, маркетинг без содержания, финансовые новости без продуктовой части.
- **Если после полного прохода < 5 кандидатов** — расширенный проход: «LLM news <дата>», «AI model release <дата>», «AI coding agent update <неделя>». Пустой день должен быть доказан широким поиском, а не одним запросом.

Дедупликация: URL уже в `seen_urls` (т.е. опубликован ранее) → пропусти сам URL, но НЕ сюжет. **Продолжение известного сюжета из `clusters` с новым содержанием** (новые факты, цифры, релиз после анонса, волна реакций) — ВКЛЮЧАЙ как новость с существующим `cluster_id`: недельный срез сам склеит. Пропускается только перепечатка без новизны. Новый сюжет → новый `cluster_id` (kebab-case: `<компания>-<суть>`).

Отбери максимум `limits.max_news_items` самых значимых.

## Шаг 2. Обогащение каждой новости

Для каждой новости собери (WebSearch/WebFetch):
- **summary** — 2–3 предложения: что именно появилось и что это даёт.
- **opinions** — 2–4 реакции с РАЗНЫМИ позициями (positive / skeptical / neutral) от сообщества и экспертов, каждая с точным URL (тред HN, пост Reddit, статья). Указывай автора/площадку. Не выдумывай цитаты — только реально найденное; если реакций ещё нет, оставь пустой список.
- **use_cases** — 1–3 конкретных кейса применения со ссылками (из обсуждений, блогов, документации).

## Шаг 3. Ранжирование — impact_score (главное в продукте)

Оцени каждую новость по рубрике, выставь **impact_score 1–10** и одну строку **impact_reason**:

| Балл | Критерий |
|---|---|
| 9–10 (paradigm) | Меняет саму парадигму использования LLM; через год так будут работать все |
| 7–8 (major) | Заметно меняет существующие подходы ИЛИ открывает новый класс сценариев |
| 4–6 (notable) | Полезное развитие в рамках текущих подходов; влияет на выбор инструментов |
| 1–3 (minor) | Инкрементальное улучшение, подходы не меняет |

Взвешивай: (a) меняет ли текущие подходы, (b) открывает ли новые, (c) широта применимости (все пользователи LLM или ниша), (d) сила реакции сообщества. Оценивай консервативно: 9–10 — редкость. Для продолжения сюжета оценка может расти/падать относительно вчерашней — срезы возьмут максимум по кластеру.

## Шаг 4. Экспертный контент

Пройди **всех** экспертов из конфига (не останавливайся на первых найденных) + WebSearch «best/popular LLM article/video this week»:
- **МИНИМУМ `limits.min_expert_total` материалов (статьи+видео суммарно) в дневной сводке.** Если эксперты из конфига дали меньше — расширяй поиск: другие заметные авторы/каналы, резонансные материалы недели о подходах к использованию LLM. Максимумы: `limits.max_articles_per_day` статей и `limits.max_videos_per_day` видео.
- **Видео-discovery**: кандидатов ищи WebSearch-ом («site:youtube.com <имя канала>», «<эксперт> new video <месяц год>») и/или `yt-dlp --flat-playlist` по каналам (youtube.com и googlevideo.com добавлены в trusted domains облака). Отбирай недавнее и резонансное о ПОДХОДАХ к использованию LLM (не новости-пересказы).
- **Формат страницы конспекта (`docs/notes/{articles,videos}/<slug>.html`) — связный ТЕКСТ, а не пункты.** 3–6 абзацев с подзаголовками по смыслу, кратко рассказывающих СУТЬ материала своими словами. Если материал посвящён методу/приёму — метод ОБЯЗАТЕЛЬНО раскрыть: что это, из каких шагов состоит, когда и зачем применять, ограничения. Никаких «ключевые тезисы: 1,2,3» и отдельных таблиц таймкодов.
- **Таймкоды видео — инлайн в тексте конспекта**: в момент, когда текст пересказывает фрагмент, ставится ссылка-метка вида `<a href="https://youtube.com/watch?v=ID&t=<sec>">[MM:SS]</a>`. Кадры — `<figure>` с подписью прямо в тексте, рядом с абзацем, к которому относятся.
- **Статья:** WebFetch полного текста → нарративный конспект; схемы — inline-SVG с пояснением в тексте. Каркас HTML (шапка, `../note.css`, `../note.js`, двуязычные `<span data-lang="ru|en">`) — как у страниц за 2026-07-09 (образец формата).
- **Видео:** `python scripts/gemini_video.py <url> --out /tmp/summary.json` (тезисы/таймкоды — сырьё для текста); кадры: `python scripts/extract_frames.py <url> --from-summary /tmp/summary.json --out-dir docs/media/frames/<slug>`. Кадры пробуй ВСЕГДА (yt-dlp/imageio-ffmpeg из pip, YouTube в облаке разрешён); не получилось — конспект без кадров с пометкой, статус в run_log. Если GEMINI_API_KEY недоступен — конспект по субтитрам (`yt-dlp --write-auto-sub`).
- Slug — `scripts/common.py: slugify(en-заголовок)`.

## Шаг 5. Запись данных

Собери `data/daily/<TARGET_DATE>.json` строго по схеме (все текстовые поля — `{"ru": ..., "en": ...}`, инвариант 2):

```json
{
  "date": "YYYY-MM-DD",
  "generated_at": "ISO-8601 UTC — РЕАЛЬНЫЙ момент сборки (дашборд показывает его как «обновлено»; не ставь приблизительное время)",
  "news": [{
    "id": "<date>-<slug>", "cluster_id": "<сюжет>", "models": ["<tag из sources.yaml>"],
    "title": {"ru", "en"}, "summary": {"ru", "en"},
    "impact_score": 1-10, "impact_reason": {"ru", "en"},
    "source_url": "...",
    "opinions": [{"author", "stance": "positive|skeptical|neutral", "quote": {"ru", "en"}, "url"}],
    "use_cases": [{"text": {"ru", "en"}, "url"}]
  }],
  "expert_content": [{
    "id", "type": "article|video", "author", "title": {"ru", "en"}, "url",
    "summary": {"ru", "en"}, "note_page": "notes/<...>.html",
    "cover": "media/frames/<slug>/<файл>.jpg"  // у видео: самый показательный кадр (обложка карточки); опционально
  }]
}
```

Без поля `demo`. Обнови `data/state.json` (новые URL и кластеры). Демо-файлы (`demo: true`) при первом реальном прогоне удали из `data/daily/` и `docs/data/daily/`, демо-страницы `demo-article.html`/`demo-video.html` оставь — на них ссылается документация.

## Шаг 6. Билд и проверки

```
python scripts/build_site.py        # index.json + копия в docs/data/
python -m pytest tests/unit -q     # должны проходить
```
Валидность JSON проверь чтением; убедись, что у каждой новости есть impact_score, source_url и оба языка.

## Шаг 6.5. Отчёт о сборе (run log)

Запиши `data/run_log/<TARGET_DATE>.json` — наблюдаемость прогона (коммитится, на сайт не копируется). ВСЕГДА перезаписывай файл данными СВОЕГО прогона, даже если он существует и даже если новостей не добавилось: run_log — это след конкретного запуска, и его коммит обязателен в любом исходе. Прогон, не оставивший коммита, неотличим от упавшего:

```json
{
  "date": "...", "generated_at": "...",
  "sources": [
    {"source": "anthropic blog", "status": "ok|blocked|empty|error", "candidates": 3, "note": "..."}
  ],
  "totals": {"candidates": 18, "published": 7, "rejected": {"перепечатки": 6, "слухи": 3, "не по теме": 2}},
  "video_pipeline": {"gemini": "ok|error: ...", "frames": "ok|skipped: youtube blocked"}
}
```

Перечисли ВСЕ источники, которые обходил (модели, агрегаторы, HN API, эксперты). Если ≥1/3 источников blocked/error — обязательно скажи об этом в финальном отчёте сессии: это сигнал деградации окружения, а не «тихий день».

## Шаг 7. Публикация

Коммит: `digest: сводка за <TARGET_DATE>` (+ Co-Authored-By по конвенции). В облачной routine — `git push`; локально — спроси пользователя, пушить ли. При пересборе коммить обновлённый файл так же; если после всех шагов коммитить действительно нечего — не выходи молча, а явно объясни в финальном отчёте, почему (что искал, что нашёл, почему отфильтровал).

## Ограничения

- Не выдумывай новости, цитаты и ссылки. Нет уверенности в реальности события — не включай.
- Мало событий за день — это нормально: лучше 3 настоящих новости, чем 12 натянутых.
- Пиши двуязычный контент сразу, не «переведу потом».
